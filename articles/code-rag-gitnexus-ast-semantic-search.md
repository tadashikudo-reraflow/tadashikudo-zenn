## はじめに

Claude Code に「ローカルのコードベースを意味で検索する」MCP を入れている人は増えてきた。私の環境でも bge-m3 埋め込み + sqlite-vec のローカル semantic 検索 MCP を運用していて、33 リポジトリ・約 1,200 ファイル・5,190 チャンクをインデックスしている。「認証まわりの処理どこだっけ」「リトライ実装してる箇所」みたいな曖昧な問いには、grep より圧倒的に速く当たりを付けられる。

ただ、semantic 検索だけだと**どうしても越えられない壁**がある。

> 「この関数の引数を変えたら、どこが壊れる？」

これは「意味が似たコード」をいくら集めても答えられない。必要なのは**呼び出し関係の正確なグラフ**であって、埋め込みの近傍ではない。この記事では、semantic 検索（意味で「どこ」を見つける）と AST 依存グラフ（構造で「何が繋がっているか」を辿る）を**二刀流**で使うと、Claude Code のコード理解がどう変わるかを書く。

## semantic 検索が得意なこと・苦手なこと

埋め込みベースの検索は「自然言語の曖昧な意図」と「コードの意味」をマッチさせるのが本質的な強みだ。

| 問い | semantic 検索 | 理由 |
|------|:---:|------|
| 「決済のwebhook処理どこ？」 | ◎ | 意図と実装の意味が近い |
| 「似たようなバリデーション他にもある？」 | ◎ | 埋め込みの近傍探索そのもの |
| 「この関数を消したら何が壊れる？」 | ✕ | 呼び出し元は「意味が似てる」とは限らない |
| 「ログイン処理は結局どんな順で動く？」 | △ | チャンク断片は返るが実行順は復元できない |

最後の 2 つが弱点だ。`deleteUser()` を呼んでいる箇所は、コードの見た目も意味もバラバラで、埋め込み空間では近くにいない。**「依存」は意味ではなく構造の問題**だから、ベクトル検索の管轄外なのだ。

## AST 依存グラフという相棒

そこで使うのが [GitNexus](https://www.npmjs.com/package/gitnexus)（`npm i -g gitnexus`）のような、ソースを AST 解析して**コード知識グラフ**を構築するツール。私の環境では Claude Code に MCP として接続している。

仕組みはこうだ。リポジトリを解析すると、関数・クラス・メソッド・ファイルが**ノード**になり、`CALLS` / `IMPORTS` / `EXTENDS` / `IMPLEMENTS` といった関係が**エッジ**になる。実際に手元の中規模リポジトリ（248 ファイル）を解析すると、

- ノード 6,337
- エッジ 13,537
- コミュニティ（機能のまとまり）392
- プロセス（実行フロー）300

というグラフが出来上がった。これは sqlite-vec のチャンク集合とはまったく別の構造データだ。

このグラフに対して、たとえば「`deleteUser` を呼んでいる関数」はこう書ける（MCP 経由で Cypher を投げる）：

```cypher
MATCH (a)-[:CodeRelation {type: 'CALLS'}]->(b:Function {name: "deleteUser"})
RETURN a.name, a.filePath
```

埋め込みの近傍探索では絶対に出てこない、**100% 正確な呼び出し元リスト**が返る。

## 二刀流のワークフロー

実際に Claude Code に作業させるときは、2 つを役割分担させると噛み合う。

### ステップ1: semantic 検索で「入口」を見つける

```
（自然言語）「ユーザー削除のエンドポイントを直したい」
   ↓ semantic 検索 MCP
deleteUser() が src/services/user.ts にあると判明
```

曖昧な意図から具体的なシンボル名・ファイルに着地する。ここは埋め込みの独壇場。

### ステップ2: 依存グラフで「影響範囲」を確定する

入口のシンボルが決まったら、グラフ側の `impact` で blast radius を測る。GitNexus の impact ツールは影響を**深さ別**に返してくれる：

- `d=1`: 直接の呼び出し元 → **必ず壊れる**
- `d=2`: 間接的に影響 → 要確認
- `d=3`: 推移的 → テスト推奨

さらに risk（LOW / MEDIUM / HIGH / CRITICAL）と、**どの実行フロー（プロセス）が何ステップ目で壊れるか**まで返る。Claude Code はこれを読んで「d=1 の 3 箇所を先に直し、UserLogin プロセスのステップ4 が影響を受けるのでそこのテストを足す」といった**根拠のある計画**を立てられる。

### ステップ3: context で 360 度ビュー

個別シンボルを深掘りするときは `context`。呼び出し元・呼び出し先・実装インターフェース・参加している実行フローをまとめて返すので、「この関数は何者で、どこから来てどこへ行くのか」が一望できる。

```
semantic（どこ？）→ impact（何が壊れる？）→ context（この子は何者？）
```

この順で回すと、**「意味で当たりを付け、構造で裏を取る」**という人間の熟練エンジニアに近い読み方を Claude Code が再現できる。

## なぜ「片方だけ」では足りないのか

「グラフがあるなら semantic 検索いらないのでは？」と思うかもしれないが、逆も真ではない。

- **グラフだけ**だと、最初のシンボル名を人間が知っている前提になる。「決済まわり」みたいな曖昧な入口から `Function {name: ...}` を特定できない。Cypher は名前を知らないと書けない。
- **semantic だけ**だと、前述のとおり依存・実行順が出ない。リファクタリングの安全確認ができない。

両者は**インデックスの種類が根本的に違う**（ベクトル近傍 vs グラフ走査）ので、片方がもう片方を包含することはない。だからこそ二刀流に意味がある。コストも、どちらもローカルで完結する（埋め込み計算 + SQLite、AST 解析 + 組み込みグラフDB）ので、外部 API 課金はゼロだ。

## まとめ

- **semantic コード検索**は「曖昧な意図 → 具体的なシンボル」が爆速。だが依存・実行順は出せない。
- **AST 依存グラフ**は「この変更で何が壊れるか」を 100% 正確に出せる。だが入口のシンボル名を知らないと使えない。
- 二刀流の黄金ルートは **semantic（どこ？）→ impact（何が壊れる？）→ context（何者？）**。
- どちらもローカル完結で API 課金ゼロ。Claude Code の MCP として両方接続すると、コード理解が「検索」から「設計の裏取り」へ一段上がる。
- まず semantic 検索を入れていて物足りなさを感じているなら、次の一手は依存グラフ系 MCP の追加が効く。
