# Claude Codeで本番コードに触れる前に、まずロールバック手順を書け——Rollback path先記述の設計

## はじめに

AIエージェントに本番コードを触らせるとき、あなたはどんな順番で指示を出しているだろうか。

「このAPIエンドポイントを修正して」「DBスキーマを更新して」——こういった指示をClaude Codeに渡すとき、多くのエンジニアは **実装内容から書き始める**。ゴールを伝え、コードを生成させ、レビューして、マージする。

しかし、筆者がAIエージェント運用で痛感したのは別の順番だった。

**「rollbackが曖昧ならagent境界も曖昧」**

ロールバック手順が書かれていない実装プロンプトは、エージェントの「どこまでやっていいか」という境界線も曖昧にする。結果として、予期しない副作用・本番障害・手動復旧の三重苦に陥る。

この記事では、Claude Codeを使った本番隣接コード変更における **「Rollback path先記述ルール」** の設計思想と具体的な適用パターンを紹介する。

---

## なぜ「rollback後書き」では遅いのか

### エージェントは「やり直し」を想定しない

通常のエンジニアなら、実装しながら暗黙的に「ここが失敗したらgit resetすればいい」「このマイグレーションはDROP TABLEで戻せる」と頭の中でロールバックパスを計算している。

しかしAIエージェントにはその暗黙知がない。エージェントは **指示されたゴールを達成することに最適化** されており、「失敗した場合の復旧」については指示がない限り考慮しない。

たとえば、次のようなプロンプトを渡したとしよう。

```
Stripeのwebhookハンドラーを修正して、サブスクリプション更新イベントのとき
plan_typeカラムを'premium'に更新するようにして。
```

エージェントはこの指示を忠実に実行する。しかし以下の点は考慮されない可能性がある：

- 本番DBに直接マイグレーションをかけた場合の復旧手順
- Webhookのシグネチャ検証がスキップされたときの影響
- plan_typeが想定外の値のまま残った場合のデータ整合性

ロールバック手順を **後から** 考えようとしても、その時点ではすでにエージェントが変更を完了している。

### スペック先行ルールとの組み合わせ

筆者の開発環境では、以下の変更カテゴリに対して「スペック.md先行ルール」を適用している：

1. **データ構造変更**（DBスキーマ / pgvector index / 共通型定義）
2. **RAGインデックス影響**（検索エンジンの挙動を変える変更）
3. **複数PJ波及**（2つ以上のサービスに影響する変更）
4. **決済フロー**（Stripe checkout / webhook / portal）

これらの変更は「ロールバックコストが高い変更」と定義しており、実装前に必ずspec.mdを作成するルールだ。

そしてこのスペック先行ルールに、2026年5月から **Rollback path先記述ルール** を追加した。

**spec.mdを書くとき、ロールバック手順を冒頭のセクションに置く。**

---

## Rollback path先記述の具体的なフォーマット

実際に使っているspec.mdの冒頭テンプレートはこうなっている：

```markdown
# spec: [変更名] YYYY-MM-DD

## ロールバック手順（実装前に必ず記入）

### 前提条件
- [ ] 本番DBのスナップショット取得済み: `pg_dump -U postgres appdb > backup_YYYYMMDD.sql`
- [ ] 影響を受けるサービスのバージョン確認: `git tag --points-at HEAD`

### ロールバック手順
1. **Stripe webhook停止**: Stripe DashboardでWebhook endpointを無効化
2. **コードのリバート**: `git revert HEAD --no-edit && git push origin main`
3. **DBマイグレーション逆適用**: `psql appdb < backup_YYYYMMDD.sql`
4. **Stripe webhook再有効化**: エンドポイントを再度有効化
5. **動作確認**: テスト用Stripe CLIイベントで正常動作確認

### ロールバック判断基準
- 本番エラーレートが通常の3倍超過
- 決済成功率が95%を下回る
- 5分以内に原因特定できない場合は即ロールバック

---
## 要件
...
```

ポイントは **「ロールバック手順を先に埋めないとspecが完成しない」** という構造にすることだ。後付けで「まあ大丈夫でしょ」と飛ばせないように、冒頭に置く。

## エージェントへの渡し方

spec.mdを書いたら、Claude Codeへの指示はこうなる：

```
以下のspec.mdに従って実装を進めてください。

【重要】実装前にspec.mdのロールバック手順を読み、
各ステップが実行可能か確認してください。
不明な点があれば実装を開始する前に質問してください。

spec.md:
---
# spec: Stripe webhook plan_type更新 2026-05-15

## ロールバック手順（実装前に必ず記入）
...
```

このように渡すと、エージェントは実装の「境界線」を自然に認識する。

- **どこまでDBを変えていいか**（スナップショットが前提なので、バックアップなしのDDLは止まる）
- **どこで失敗したら止まるべきか**（判断基準が明示されているため）
- **自分では判断できない事項**（Stripe DashboardのUI操作はエージェントにできないため、必ず確認を求める）

## plan-reviewとの組み合わせ

筆者の環境では `/plan-review` スキルを使って実装計画をレビューしているが、このスキルのチェックポイントの一つ（P3）が **「ロールバック手順が明記されているか」** だ。

```
P3: 本番隣接変更のロールバック手順は spec.md 冒頭に記述されているか？
    - DB変更の場合: pg_dump コマンドが具体的に書かれているか
    - コード変更の場合: git revert / git reset の対象ブランチが明確か
    - 外部サービスの場合: Stripe/Cloudflare等の手動操作ステップが含まれるか
```

spec.mdなしでエージェントを起動しようとすると、このチェックで引っかかる設計になっている。

## 実際に助かったケース

あるSaaSプロジェクトで、Next.jsのルーティング設定（`next.config.ts`）の redirect ルールを変更したときのことだ。

最初は「この redirectを削除して、代わりに app/ 配下に直接ページを作る」という指示を出した。エージェントは指示通り実装した。しかしデプロイ後、旧URLへのリンクが全滅した。

ロールバック手順を先に書いていれば：

```markdown
## ロールバック手順
1. `git revert HEAD --no-edit` で redirect削除をリバート
2. `git push origin main` でデプロイ
3. 旧URL（/old-path）へのアクセスが301リダイレクトで復活することを確認
```

この手順があれば、**実装前に「旧URLへのリンクは全部どこかに書いてあるか？」** という質問が自然に浮かぶ。ロールバック手順を書く行為自体が、**実装の抜け漏れを発見するトリガー** になる。

## ルールの適用範囲

すべての変更にこのルールを適用するのは過剰だ。以下の判断基準で絞っている：

| 変更カテゴリ | ロールバック先記述 | 理由 |
|------------|-----------------|------|
| 本番DBスキーマ変更 | **必須** | データ損失リスク高 |
| 決済フロー変更 | **必須** | 課金影響が即金銭損失 |
| 複数サービス波及 | **必須** | 1箇所のリバートが他サービスを壊す |
| RAGインデックス変更 | **必須** | インデックス再構築に時間がかかる |
| フロントエンドUIのみ | 任意 | git revertで即戻せる |
| ドキュメント・記事 | 不要 | 変更が外部公開文章のみ |

「失敗したとき、5分以内に元に戻せるか？」がYesなら先記述は省略できる。Noならルールを適用する。

---

## まとめ

Rollback path先記述ルールの実践ポイントをまとめる：

- **ロールバック手順はspec.mdの冒頭に置く**: 後付けではなく、実装計画より前に書く構造にする
- **エージェントへの指示にspec.mdを丸ごと渡す**: ロールバック手順があることでエージェントの「境界感覚」が変わる
- **書く行為自体が抜け漏れ検出になる**: 手順を書きながら「あれ、これどうする？」が自然に出てくる
- **適用範囲を絞る**: すべてに適用すると形骸化する。「5分で戻せるか」を判断基準にする
- **plan-reviewのチェックポイントに組み込む**: 人間のレビューだけでなく、スキルのゲートとして機能させる

「rollbackが曖昧ならagent境界も曖昧」——これはAI時代の本番運用における一つの原則として、チームのルールに組み込む価値があると感じている。

エージェントに任せるほど、人間側の設計の解像度が問われる。
