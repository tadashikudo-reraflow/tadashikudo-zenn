# Claude Codeの出力がHTMLになった理由——Markdown→AI→HTML→人間という設計

## はじめに

Claude Codeに指示を書くときはMarkdownを使う。`CLAUDE.md`、仕様書、メモ、設計ノート——どれもMarkdownだ。

ところが、Claude Codeが生成するドキュメントやレポートを見ていると、気づいたらHTMLファイルが増えていた。

- `ops-flow.html`（運用フロー全体図）
- `architecture.html`（4-Agentアーキテクチャ可視化）
- `model-history.html`（モデル比較・履歴）
- `options-comparison.html`（技術選定の比較表）

なぜ出力だけHTMLになるのか。「AIがMarkdownを読んでHTMLを出す」という非対称な設計には、理由がある。

---

## 入力がMarkdownである理由

Markdownは「人間が書くためのフォーマット」として、AIとの相性が非常に良い。

**1. AIが構造を理解しやすい**

Markdownのヘッダー（`#`, `##`）・リスト・コードブロックは、自然言語に近い構造を持つ。Claude Codeが`CLAUDE.md`を読み込むとき、セクション境界を把握しながらコンテキストを構築できる。XMLのような厳密な閉じタグも、JSONのようなネストも不要だ。

**2. 人間が書き続けやすい**

仕様書や設計ノートをYAMLやHTMLで書き続けるのは辛い。Markdownならエディタを問わず書けて、差分もGitで管理しやすい。AIへの「指示書」はメンテナンスコストが低いほど良い。

**3. バージョン管理が自然に機能する**

```
spec-2026-05-01.md
spec-2026-05-15.md（前者を更新）
```

テキストベースなので、`git diff`で変更点が一目でわかる。AIへの指示の変遷をそのままトレースできる。

---

## 出力がHTMLになる理由

問題は「AIが生成したドキュメントを人間が読む」という段階だ。ここでMarkdownの制約が表面化する。

**Markdownでは表現できないレイアウトがある**

たとえば「4つのAIエージェントの役割比較」を設計ドキュメントに落とす場合を考える。

Markdownで書けるのはこの程度：

```markdown
## Claude Code
- 役割: メイン開発AI
- 得意: コード編集・ファイル操作

## Codex
- 役割: バッチ処理
- 得意: 並列実行・テスト生成
```

これで情報は伝わるが、**視覚的な比較が難しい**。4つを横並びにしたい、色分けしたい、タブで切り替えたい——そういった要求にMarkdownは答えられない。

HTMLなら以下が実現できる：

```html
<div style="display:grid;grid-template-columns:repeat(4,1fr);gap:16px">
  <div class="agent-card claude-card">
    <h3>Claude Code</h3>
    <p class="role">メイン開発AI</p>
    <ul>
      <li>コード編集</li>
      <li>ファイル操作</li>
    </ul>
  </div>
  <!-- 他3つのエージェント... -->
</div>
```

ブラウザで開くだけで、色分けされたグリッドレイアウトが表示される。

## 実際の運用パターン

筆者の環境では、複数のHTMLファイルが「人間が読むためのドキュメント」として機能している。

### パターン1: アーキテクチャ可視化

```
仕様書（Markdown）→ AI分析 → architecture.html（タブ付きHTML）
```

4つのランタイム（Claude Code / Codex / Antigravity / Cursor）の役割分担を、タブUIで切り替えながら確認できるHTML。MCP接続状況や権限マトリクスも一覧できる。

Markdownだと10スクロールが必要な情報が、HTMLなら1画面に収まる。

### パターン2: 運用フロー図

```
PJ仕様（Markdown）→ AI設計 → ops-flow.html（フロー図）
```

AIサービスの運用パイプラインを視覚化したHTML。データ収集→前処理→予測→通知→記録の流れを、色分けされたフローチャートで表現している。

`ops-flow.html`はPJが進化するたびに更新され、バージョン番号（v1, v2, v3...）が記録される。Markdownの散文よりも、変更箇所が一目でわかる。

### パターン3: 技術選定の比較表

```
要件定義（Markdown）→ AI調査 → options-comparison.html（比較マトリクス）
```

複数の技術選択肢を縦横に並べた比較表。セルに色を付けて「◎ / ○ / △ / ✗」を視覚化できる。Markdownテーブルでも書けるが、10列×5行を超えると崩れやすい。

## 設計原則として整理する

この実践から導き出せる原則がある。

```
書く人（人間） → Markdown → AIが読む → 処理 → HTML → 人間が読む
```

**Markdownは「AIとの対話言語」**として機能する。AIが構造を理解しやすく、人間が書き続けやすい。

**HTMLは「人間への出力言語」**として機能する。ブラウザという普遍的なビューアがあり、追加のツールなしで複雑なレイアウトを表現できる。

この非対称性は意図的だ。AIと人間では「読みやすさ」の定義が違う。

| 対象 | 好むフォーマット | 理由 |
|------|----------------|------|
| AI（Claude等） | Markdown / プレーンテキスト | トークン効率・構造理解のしやすさ |
| 人間 | HTML / PDF | 視覚的レイアウト・色・インタラクション |

---

## いつMarkdownのままにするか

ただし、すべてをHTMLにする必要はない。

**Markdownのままで良いケース：**
- AIが再度読み込む可能性があるドキュメント（仕様書、CLAUDE.md）
- Git差分が意味を持つドキュメント（ADR、更新履歴）
- 数段落の説明で済む内容（TODO、メモ）
- CIや自動処理がパースする設定ファイル

**HTMLにすべきケース：**
- 人間だけが読む最終成果物（プレゼン用の設計図）
- 複数の情報を並べて比較する（選定比較、モデル評価）
- 進捗や状態を継続的に可視化する（ダッシュボード、フロー図）
- 外部ステークホルダーに共有する（レポート、サマリー）

---

## HTMLをGit管理する際の注意点

「HTMLはバイナリではないからGit管理できる」——その通りだが、いくつか気をつけることがある。

**1. 生成コードはコンテンツとして扱う**

AIが生成したHTMLは「コード」ではなく「コンテンツ」として分類する。ソースコードリポジトリに混入させず、コンテンツ専用のディレクトリに置くと管理しやすい。

```
contents/
  PJ33/
    ops-flow.html     ← AIが生成した可視化HTML
    model-history.html
  _shared/
    architecture.html  ← 複数PJで参照する共通HTML
```

**2. インラインスタイルで依存を減らす**

外部CSSやJavaScriptライブラリへの依存があると、オフライン時や長期保存時に壊れる。スタイルはインライン or `<style>`タグに閉じ込めるのが安全だ。

**3. バージョン番号を本文に入れる**

```html
<p class="version">v3.1 | 2026-05-20更新</p>
```

ファイル名でのバージョン管理（`ops-flow-v3.html`）は増殖するので、ファイル名は固定しつつ本文にバージョンを書く方式が扱いやすい。

---

## まとめ

Claude Codeとの実践から見えてきた設計原則：

- **入力はMarkdown**: AIが読みやすく、人間が書き続けやすい唯一のフォーマット
- **出力はHTML**: ブラウザという普遍的なビューア+複雑なレイアウト表現が目的なら迷わずHTML
- **この非対称性は意図的**: AIと人間では「読みやすさ」の定義が違う
- **HTMLはコンテンツとして管理**: コードリポジトリではなくコンテンツ専用ディレクトリへ
- **再利用されるものはMarkdown、見られるだけのものはHTML**: 用途で使い分ける

「AIに書かせたドキュメントが増えてきたがMarkdownでは読みにくい」と感じたら、HTMLへの移行を検討してみてほしい。ブラウザさえあれば追加のツールは不要だ。
