# 104kスターのKarpathy CLAUDE.mdを試したら、手元に既にあった話——グローバルCLAUDE.mdで全PJ適用済みの実態

## はじめに

Andrej Karpathy氏が公開した `CLAUDE.md` テンプレートがGitHubで伸びていて、タイムラインでも「入れた方がいい」「エラーが減った」という声をよく見るようになりました。筆者も「コピペしようかな」と眺めていたのですが、**よく読むと書かれている原則のほとんどが、すでに自分のグローバル `~/.claude/CLAUDE.md` に実装されていた**ことに気づきました。

この記事では、Karpathy系で語られている原則／グローバル `CLAUDE.md` として運用するメリット／実際に効いている**5つのコア原則**を整理します。「PJごとに `CLAUDE.md` をコピペして疲れてきた人」向けです。

---

## Karpathy系 CLAUDE.md が言っていること

最近の「ちゃんと動く」 `CLAUDE.md` に共通しているのは、**プロンプト芸ではなくシニアエンジニアの行動規範を文章化したもの**だという点です。よく出てくる項目をまとめると、

1. **小さく正確な変更を好む（Surgical Changes）** — "ついで改善" 禁止
2. **既存スタイルに合わせる** — 自分の好みを持ち込まない
3. **dead codeを勝手に削除しない** — 自分の変更で生まれた孤児だけ消す
4. **「動いた気がする」で終わらせない** — 検証して初めて完了
5. **エラーは最低3つのアプローチを試してから報告する**
6. **複雑なタスクは先に計画を立てる**（3ステップ以上）
7. **仕様が曖昧なら先に質問させる**（"Let Claude interview you"）

要は「**勝手に賢くなろうとさせない**」「**幻覚で進ませない**」ためのガードレールです。

## なぜ「グローバル CLAUDE.md」に置くのか

Claude Code の `CLAUDE.md` は階層的に読み込まれます。グローバル（`~/.claude/CLAUDE.md`）→ プロジェクト直下 → サブディレクトリ、の順にマージされます。

筆者の運用は「**コア原則はグローバルに置き、PJローカルは差分だけ**」です。理由は単純で、PJごとにテンプレをコピペすると**ドリフト**する（コピー先で書き換えられて原則が壊れる）／グローバルに置けば**新規PJを始めた瞬間から行動規範が効く**／PJ側はそのPJ固有の知識（DBスキーマ・デプロイ手順・禁止操作）だけに集中できる、の3点です。

## 実際に効いている5つのコア原則

自分の `~/.claude/CLAUDE.md` 「コア原則」セクションは抜粋するとこうです。

```markdown
- **小さく正確な変更**を好む。影響範囲は常に最小限
  - 隣接コード・コメント・フォーマットを"ついでに改善"しない
  - 既存のdead codeは削除せず指摘のみ。自分が生んだ孤児だけ削除
- 一時しのぎを嫌う。**根本原因を突き止め、正しく直す**
- 「動いた気がする」では終わらない。**検証して初めて完了**
- エラー発生時は**最低3つの異なるアプローチ**を試してから報告
- 複雑なタスクは**まず計画を立てる**（3ステップ以上 → プラン・モード）
- 実装前に**仮定とトレードオフを明示**。サイレントな意思決定をしない
- **仕様が曖昧なら先に質問**: いきなり実装に入らない
```

### 原則1: Surgical Changes（ついで改善禁止）

地味ですが**一番効きます**。バグ修正依頼に「ついでにリファクタしました」と巨大diffが返ってくる、あれを止める条文です。ポイントは**禁止対象を具体名で書く**こと。「隣接コード」「コメント」「フォーマット」「dead code」まで列挙してあります。「リファクタするな」だけでは曖昧で守られません。

### 原則2: 「検証して初めて完了」

型エラーが消えただけで「動作確認しました」、`npm run build` が通っただけで「デプロイ準備完了です」を防ぎます。文面は一行ですが、**ユーザー側が「検証なしの完了宣言を毎回指摘する」運用とセット**ではじめて効きます。

### 原則3: 3アプローチ・ルール

> エラー発生時は**最低3つの異なるアプローチ**を試してから報告

「動きません」だけで投げ返してくるのを防ぎます。運用してみると、**1個目で諦めていた問題の8割は2〜3個目で自走解決**します。逆に「3つ試してダメ」と言われたら、ユーザー側も本気で見るスイッチが入ります。

### 原則4: プラン・モード強制（3ステップ以上）

`CLAUDE.md` には「複雑なタスクは**まず計画を立てる**（3ステップ以上 → プラン・モード）」と一行書き、実体は Claude Code の `/plan` モードに乗せます。「3ステップ」という**数値の閾値**を明示するのがミソです。「複雑なら」だけでは毎回スキップされます。

### 原則5: 仕様が曖昧なら質問させる

Karpathy系で一番話題になった項目で、「**仕様が曖昧なら先に質問**: いきなり実装に入らず、要件を質問させてから着手」と書くだけで「ヒアリングから始める」モードになります。書いてないと、いきなり実装に飛び込んで想定と違う実装が出来上がる事故が起きます。

## やってみてわかった運用Tips

### Tip 1: タグ凡例で「絶対」と「推奨」を分ける

筆者の `CLAUDE.md` 冒頭はこうです。

```markdown
- **[I] Instruction**: 絶対ルール。違反時は即停止・報告
- **[G] Guidance**: 推奨指針。状況判断OK
```

ルールごとに `### [I] API課金ゲート` のようにタグ付けでき、AIに「絶対」と「推奨」を明示できます。`[I]` を乱発すると効力が下がるので、本当に止めてほしいものだけに絞ります。

### Tip 2: 本体は160〜180行・詳細は外部ファイル

公式は400行以下推奨ですが、実運用では200行を超えたあたりから「後半が無視されている気配」が出てきます。重い詳細は `~/.claude/docs/` 配下に切り出し、本体には `→ ~/.claude/docs/xxx.md` というポインタだけを残します。

### Tip 3: PJローカルは「差分」だけ書く

PJローカルの `CLAUDE.md` は、そのPJ固有のDBスキーマ・デプロイURL・触ってはいけないファイル・禁止操作だけを書きます。Surgical Changes やプラン・モードは**グローバルから降ってくる**ので再記述不要です。

### Tip 4: 月次で棚卸し

放置すると `CLAUDE.md` は太ります。月1で「30日以上更新されていない条文 × 直近で破られたルール」を洗い出し、無効化か書き直しを決めます。鮮度を保つこと自体がルールの効力を支えます。

## まとめ

- Karpathy系 `CLAUDE.md` の本質は**シニアエンジニアの行動規範の文章化**
- **グローバル `~/.claude/CLAUDE.md`** に置けば、新規PJ開始時から全原則が効く
- 実際に効く5つ: **Surgical Changes / 検証して初めて完了 / 3アプローチ / プラン・モード / 曖昧なら質問**
- `[I]` 絶対と `[G]` 推奨を**タグで区別**し、絶対ルールをインフレさせない
- 本体は**160〜180行**目標・詳細は外部ポインタ、PJローカルは**差分だけ**

「すごい新テンプレを入れる」より「**既にあるグローバルをちゃんと効かせる**」方がROIが高いです。差し替える前に、まず自分の `~/.claude/CLAUDE.md` に上の5原則が書かれているかを確認してみることをおすすめします。
