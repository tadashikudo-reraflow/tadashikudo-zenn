# Claude Code の「Spec先行ルール」：外部公開とデータ構造変更で必ず仕様書を書く理由

## はじめに

Claude Code を半年ほど業務で運用していて、最も「効いた」ルールが **Spec先行（仕様書先行）ルール** でした。

「LPの文言を直して」「DBスキーマにカラム追加して」とAIに頼むと、Claude Code は素早く実装を返してくれます。便利ですが、外部公開やデータ構造に関わる変更でこれをそのまま回すと、後で必ず痛い目に遭います。

- 公開ページの誤字・誤情報がそのままインデックスされる
- スキーマ変更に他PJのコードが追従できず、本番でクラッシュする
- RAGのチャンク形式を変えた結果、過去の検索精度が壊れる

そこで筆者の環境では、**「実装前に必ず spec-YYYY-MM-DD-xxx.md を書く」** を絶対ルール（破ったら即停止）として `CLAUDE.md` に明記しています。本稿ではその設計と運用を共有します。

## なぜSpec先行が必要か

Claude Code はセッションを跨ぐ記憶を `CLAUDE.md` とメモリファイル経由で持っていますが、それでも **「いま何を、なぜ、どの基準で正解とするか」を言語化していない依頼** は危険です。

具体的には次の3つが起きます。

1. **要件のサイレントな解釈**：「カラム追加して」と頼むと、AIが型・NOT NULL・既存データのバックフィル方針を勝手に決める
2. **検証基準の欠落**：「動いた」の判定が曖昧で、E2Eが落ちて初めて気付く
3. **波及範囲の見落とし**：共通型定義を直したらRAG chunkの整形ロジックが壊れた、など

Specを書く＝**AIに「成功条件」を渡す**ことです。Claude Code はゴール駆動が得意なので、合格基準が明確だと格段に精度が上がります。

## Spec先行ルールの対象（4カテゴリ）

筆者は「全変更にSpecを書く」運用は失敗しました。Specを書くコスト > 得られる安全性、になるからです。そこで以下の4カテゴリに **行数問わず適用** という形で絞っています。

### 1. 外部公開

LP / API / SEO記事 / SNS投稿 / 決済フロー。ユーザーや検索エンジンの目に触れる変更は、文言1行でもSpecを書きます。

### 2. データ構造変更

DBスキーマ / pgvectorのindex / RAG chunks の形式 / プロジェクト横断の共通型定義。一度本番に流すとロールバックコストが跳ね上がる領域です。

### 3. RAG index影響

ベクトル検索の挙動を変える変更（埋め込みモデル変更、chunk size変更、metadata schema変更）。検索品質はSpecがないと再現性のある評価ができません。

### 4. 複数PJ波及

2PJ以上のコード・ドキュメント・設定に影響する変更。共通ライブラリ、CI設定、ブランドカラー定義など。

## Specの最低構成

Specは長くなくていい。むしろ **A4 1枚で書ける程度の解像度** が理想です。筆者のテンプレ：

```markdown
# spec-2026-05-05-add-user-tier-column

## 1. 背景・目的
- なぜ必要か（why）。トリガーとなった事象を1行で。

## 2. 要件
- スコープに入るもの（in scope）
- スコープに入らないもの（out of scope）

## 3. 設計
- DB変更: users テーブルに tier ENUM('free','pro','enterprise') NOT NULL DEFAULT 'free'
- マイグレーション: 既存行は 'free' でバックフィル
- 影響を受けるAPIエンドポイント: /api/users, /api/billing/*
- 影響を受ける型定義: shared/types/user.ts

## 4. 検証基準（Definition of Done）
- [ ] migration up/down が冪等
- [ ] 既存テストがgreen
- [ ] 新規テスト: tier別の請求計算3パターン
- [ ] ステージングで管理画面から tier 変更が反映される

## 5. ロールバック手順
- migration down 一発で戻せること
- データバックフィル前にスナップショット取得
```

ポイントは **「検証基準（DoD）」を必ず書く** こと。ここがあるとClaude Codeに「このSpecを満たすように実装して」と言うだけで、自己検証まで回ります。

## ファイル配置と命名

横断的に使えるよう、置き場所も統一しています。

```bash
# コードリポジトリ側
~/workspace/pj/<PJ名>/docs/spec-YYYY-MM-DD-<topic>.md

# ドキュメント・ナレッジ側（Obsidian Vault等）
01_Projects/**/<PJ名>/30_docs/spec-<topic>.md
```

命名にYYYY-MM-DDを入れるのは、後から `ls docs/ | sort` で時系列に追えるようにするためです。Specは**作って終わり**ではなく、後日「あのときなぜこう決めたか」を辿る一次資料になります。

## 運用のコツ：AIにSpecを書かせる→人間がレビュー

「Spec書くのが面倒だから結局守られない」が最大の敵です。筆者は次のフローで運用しています。

```
1. ユーザー：「users に tier カラム追加したい」と依頼
2. Claude Code：実装に入らず、まず spec-*.md のドラフトを生成
3. ユーザー：Spec をレビュー（in/out scope、DoD、ロールバック手順を確認）
4. Claude Code：Spec を満たすコード＋テストを実装
5. Claude Code：DoD を1項目ずつチェックしながら自己検証
```

**「Specを書かずに実装に入ろうとしたら止める」** ためのフックを `CLAUDE.md` に書いておくと、AI側が自発的にSpec生成から始めるようになります。

```markdown
[I] Spec先行ルール
以下のいずれかに該当する変更は、実装前に必ず spec-YYYY-MM-DD-*.md を作成すること。
- 外部公開（LP/API/記事/決済）
- データ構造変更（DB schema/index/共通型）
- RAG index影響
- 複数PJ波及

Specなしで実装を始めることは禁止。違反時は即停止して報告。
```

`[I]` は Instruction（絶対ルール）の意味で、Guidance（推奨）と区別しています。

## 失敗事例から見る費用対効果

Spec先行を導入する前に経験した事故を匿名化して紹介します。

- **事故A**：SEO記事の本文をSupabase直接更新→ISRキャッシュ残存で旧版が配信され続ける。Spec化していればキャッシュ戦略が必須項目に入っていた
- **事故B**：共通型 `User` を「軽い気持ちで」リファクタ→他PJのビルドが3つ落ちた。波及範囲を Spec の「影響を受ける型定義」欄に書いていれば検知できた
- **事故C**：RAG chunk形式を変更→過去の検索結果が再現できず、社内ベンチマークが無効化

これらは全て **Specに10分かければ防げた** 事故でした。

## まとめ

- Claude Code は**ゴール駆動が強い**ので、合格基準を明文化するほど精度が上がる
- Specの対象は **「外部公開／データ構造／RAG／複数PJ波及」の4カテゴリに絞る**（広げすぎない）
- テンプレは **背景・要件・設計・DoD・ロールバック** の5項目で十分
- 運用のコツは **AIに書かせて人間がレビュー**。Specなしで実装に入らないルールを `CLAUDE.md` に明記
- Specは**過去判断の一次資料**にもなる。`spec-YYYY-MM-DD-*.md` で時系列に追えるように

「AIに任せると怖い」と感じたら、その怖さの正体は大抵 **要件と検証基準の欠落** です。Spec先行ルールはその欠落を構造的に塞ぐ、最小コストの安全装置だと感じています。
