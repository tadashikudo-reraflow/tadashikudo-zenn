# AI Agent の自己診断を信用するな：「証跡必須化」ポリシーで誤報を撲滅した話

## はじめに

Claude Code や Codex で Agent を組み合わせた自動化を組んでいると、一度はこういう経験があるはずです。

> Agent 「外部 API キーが期限切れのようなので、WebSearch にフォールバックしました」
>
> 自分 「えっ……（手元で叩く）……普通に動くじゃん」

筆者の環境で実際に起きました。Scheduled Task で動いていた SEO 記事生成 Agent が「RAG が落ちている」と報告したのですが、Docker も API キーも全部生きていた。Agent は **検索コマンドを一度も実行せずに**、別経路の関係ないエラーを「RAG 障害」と誤帰因していたのです。

この記事では、そのインシデントを起点に導入した **「Agent 自己診断 = 実機ログ証跡を必須」ルール** の設計と、SKILL.md / Eval / Scheduled Task 完了報告への落とし込みを共有します。

## 何が起きたか：サイレントフォールバックの罠

シナリオはこうでした。

1. 日次の SEO 記事生成タスクが起動
2. Agent は PostgreSQL + pgvector の RAG から関連情報を取って記事を書く設計
3. ところが Agent は「RAG が落ちているので WebSearch にフォールバック」と報告
4. 出来上がった記事は、社内 RAG にあるはずの一次情報を一切引いていない

実機検証の結果、Docker コンテナは正常稼働、API キーは有効、検索スクリプトを手で叩いたら 27 チャンクヒット。つまり Agent の診断は **完全な誤報** でした。

何が起きていたかというと、Agent は別工程で踏んだ無関係なエラー（あるモデルの旧バージョン名が deprecated になったエラー）を見て、「RAG が落ちている」と推論で結びつけていた。最悪なのは、**RAG 検索コマンド自体が一度も実行されていなかった** こと。「動かないと判断する」のと「動かないことを確認する」は別行為だ、という基本がすっぽ抜けていました。

## 根本原因：LLM の「推論で診断する」癖

LLM は文脈から原因を推論するのが得意です。これはコードレビューでは武器ですが、**インフラ診断では致命的なバイアス** になります。

人間の SRE なら、まず `curl` を叩く・`logs` を見る・`ps` で確認する、といった「実機を見る」行動から始めます。一方 LLM Agent は、

- ログに `RuntimeError` → 「全体が壊れている」と一般化
- API ドキュメントに deprecated と書いてある → 「自分の環境でもエラーになる」と推測
- 過去の似たエラーメッセージ → 「今回も同じ原因だろう」と結論

これは推論であって、検証ではない。

しかも Agent の最終報告には **もっともらしい説明** が並ぶので、人間側も「まあそういうこともあるか」と流してしまいます。

## 解決策：実機ログ証跡を必須化する

導入したルールはシンプルです。

> **「外部システムが動かない」と判断する前に、必ず実コマンドを叩いて exit code + stderr 先頭5行を取得・報告に含める**

トラストではなく、verify を強制する。Agent の自然言語報告だけでは採用しない。

### SKILL.md への落とし込み

各 Agent の `SKILL.md` に明示的なルールを書きます。

```markdown
## RAG 必須ルール

外部 RAG を参照するタスクでは:
1. RAG 検索コマンドを必ず実行する（フォールバックの前に）
2. 完了報告に `rag_search_executed` / `rag_search_chunks` / `rag_fallback_reason` を含める

WebSearch フォールバックは「RAG コマンド実行 → 0 件 or 例外」が確認された場合のみ許可。
推論によるフォールバック判断は禁止。
```

ポイントは **「フォールバック時の実エラー」も証跡で要求する** こと。「動かなかったから WebSearch にした」という曖昧報告を許さない。

### Eval スコアでの強制

SKILL.md の `Eval` セクションで、証跡なしを定量ペナルティ化します。

```markdown
## Eval

- research_score:
  - RAG 証跡あり（executed=true + chunks 記録）: 満点
  - 証跡なし: 0 点
- citation_score:
  - 一次情報リンクあり: 満点
  - 「公式情報によれば」等の曖昧引用: 0 点
```

スキル完了時に `skills-usage.json` へ自動記録するパイプラインに引っ掛けて、低スコア記事はレビューキューに回す運用ができます。

### Scheduled Task 完了報告のスキーマ化

完了報告にも構造化フィールドを足します。

```python
report = {
    "task_id": "daily-seo-writer",
    "status": "completed",
    "rag_search_executed": True,
    "rag_search_chunks": 27,
    "rag_fallback_reason": None,
}

if not report["rag_search_executed"]:
    notify_review_queue(report)
```

タスクを「report の中身」で機械的に振り分けられるようにする。運用者が全タスクの自然言語報告を読む必要がなくなります。

## 実装するときの落とし穴

### 「実行できなかった理由」も証跡対象

外部サービスが停止していてコマンド自体叩けないケースもあります。この場合は実コマンドの代わりに以下を証跡として認める:

```bash
$ docker ps | grep rag-postgres
# (no output)
```

「コマンドを叩けなかった理由」も実機ログで示す。「おそらく停止しています」は禁止です。

### Agent が証跡をでっちあげるリスク

LLM は説得力のある偽ログを生成できます。対策は **構造化フィールドで要求すること**。自然言語ではなく `{"rag_search_executed": true, "rag_search_chunks": 27}` のような JSON で出させる。さらにチャンク内容のサンプル（最初の 50 文字）も同梱させると、嘘がつきにくくなります。

理想的には、コマンド実行自体を Agent の制御下から外して、ハーネス側で実行 → 結果を Agent に渡す形がベスト。Agent は「診断する権限」を持たず、「観測結果から判断する役」に限定する。

## 導入後の効果

- フォールバック発動が体感 8 割減（推論ベースの誤発動が消えた）
- 障害診断時間が短縮（構造化フィールドを見るだけで原因が特定できる）
- レビュー対象が機械的に絞られる（証跡なしのものだけ目視確認）

デメリットは完了メッセージが少し長くなる程度で、誤報インシデント再発のコストと比べれば許容範囲です。

## まとめ

LLM Agent を本番運用する reliability ポイント:

- **推論診断を禁止し、実機ログ証跡を必須化**：「動かないと思う」と「動かないと確認した」は別物
- **SKILL.md にルールを明示し、Eval スコアで定量ペナルティ化**：自然言語ルールでは Agent は守らない
- **完了報告を構造化フィールドにする**：自然言語報告は読み飛ばされる
- **Agent から診断権限を取り上げる**：実行はハーネス、判断は Agent

「Trust but verify」は古典ですが、LLM Agent 時代は「Don't trust until verified」の強度で運用したほうが安全です。サイレントフォールバックは見つけるのが本当に難しい。
