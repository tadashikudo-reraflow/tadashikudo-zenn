# Claude CodeチームがMarkdownにHTMLを持ち込んだ理由——そして自分はHTMLすら読まないという現実

## はじめに

Claude Codeの設定ファイル`CLAUDE.md`はその名のとおりMarkdown形式です。プロジェクトのルール・コーディング規約・AIへの指示をMarkdownで書き、Claude Codeが自動的に読み込んで動作するという設計は、シンプルで直感的です。

しかし最近のリリースノートに、ひっそりとこんな変更が入っていました。

> **Changed CLAUDE.md HTML comments (`<!-- ... -->`) to be hidden from Claude when auto-injected. Comments remain visible when read with the Read tool.**

Markdownファイルの中に、HTMLコメント専用の「Claudeには見えない層」が作られたのです。

なぜそんな設計になったのか。そして、その機能を知った自分はどう使っているか。オチをさきに言ってしまうと、**「書いたHTMLコメントを自分で読み返していない」** という情けない現実があります。

---

## CLAUDE.mdにHTMLコメント層が追加された背景

CLAUDE.md はAIへの指示書として機能します。Claudeはセッション開始時にこのファイルを自動的に読み込み、そこに書かれたルールに従って動作します。

問題は、CLAUDE.md が「AIへの指示」と「開発者向けのメモ」の両方を兼ねてしまいがちなことです。

例えば、こんな記述をCLAUDE.mdに書いたとします。

```markdown
## コーディング規約
- ESLintの設定に従うこと
- TypeScriptの`any`型は原則禁止

<!-- 2026-03-01: any型禁止は既存コードに40箇所の違反あり。
     移行期間として3ヶ月猶予を設けた経緯あり。完了目処は6月末。 -->
```

従来は、このHTMLコメントもClaudeのコンテキストに含まれていました。つまり、開発者間のメモ書きがそのままAIへの「指示」として解釈されるリスクがありました。

今回の変更で、`<!-- ... -->`内のテキストはClaude自動注入時にはスキップされ、人間が`Read`ツールで明示的に読む場合にのみ表示されるようになりました。

---

## 実用的な使い方：3つのユースケース

### 1. 変更履歴の記録

```markdown
## API設計方針

<!-- HISTORY:
  2026-01-10: RESTからgRPCへの移行を検討→工数見積もりで断念（ktd）
  2026-02-28: GraphQL採用を決定（PO承認）
  2026-04-15: 一部エンドポイントはRESTのまま残す方針に変更 -->

GraphQLを採用する。ただし外部連携用エンドポイントはRESTを維持する。
```

Claudeには「GraphQLを採用する」という指示だけが届き、なぜその判断に至ったかの経緯はコンテキストを消費しません。開発者はコメントを読めばロジックを辿れます。

### 2. 一時的なルールの無効化

```markdown
## コードレビュー規約

<!-- 以下のルールは2026-06-01まで一時停止（リリーススプリント中）
- テストカバレッジ80%以上を確認してからPRマージ
-->

PRはレビュー1名承認で即マージ可。
```

ルールを削除せず「なぜ今は適用しないか」を記録したまま、Claudeの動作だけを変えられます。

### 3. 意図・前提の注記

```markdown
## データベース設計

<!-- NOTE: このプロジェクトはNeonのFree Tierを使用。
     接続数上限20のため、Prismaのconnection_limit=5を設定済み。
     本番移行時はプール設定を見直すこと（ktd, 2026-03-20） -->

DBの接続はPrisma経由のみ使用すること。
```

「なぜこの制約があるか」という背景知識を人間向けに残しつつ、AIには「Prisma経由のみ」という簡潔なルールだけを渡せます。

---

## 逆方向の設計：WebFetchはHTMLをMarkdownに変換する

面白いことに、Claude Codeには逆方向の変換も存在します。

`WebFetch`ツールは、URLから取得したHTMLページを**自動的にMarkdownへ変換**してからClaudeのコンテキストに渡します。

これは一貫した設計思想の現れです。

- **AIが処理するテキスト** → Markdownが最適（構造が単純・トークン効率が良い）
- **人間が書くメタデータ・注記** → HTMLコメントが適切（Markdownの流儀の外側）

WebFetchが巨大なHTMLページで処理が詰まるバグが報告されたとき、修正内容は「HTML-to-Markdown変換前に入力を切り詰める」でした。これもAIへはMarkdownで渡すという原則を守った結果です。

---

## そして自分はHTMLコメントすら読まない

この機能を知って「便利だ」と思い、さっそくCLAUDE.mdのあちこちにHTMLコメントでメモを書き始めました。判断の経緯、一時的な妥協、注意事項。

しかし1ヶ月後、自分はそのコメントを一切読み返していませんでした。

エディタによってはHTMLコメントが自動で折りたたまれます。Claudeには見えないので会話の中で浮上してきません。GitのdiffでもHTMLコメントは存在感が薄い。結果として、「人間だけが読める層」を作ったつもりが、**人間も読まない層**になっていました。

これはAI活用における皮肉な問題の縮図かもしれません。AIに見せる情報の整備には力を入れるのに、自分たちが後で読むための記録は蔑ろになる。

---

## 実践的な対策

HTMLコメントを「書いたら終わり」にしないために、以下を意識するようになりました。

- **コメントに期限を書く**: `<!-- TODO: 2026-07-01までに削除 -->` のように賞味期限を設定する
- **月次でgrepする**: `grep -r "<!--" CLAUDE.md` を月次レビューのチェックリストに入れる
- **コメントが増えたら設計を見直す**: コメントが増えるのはルール設計が複雑になっているサイン

CLAUDE.mdのHTMLコメント層は強力なツールです。ただしそれを活かすには、AI向けの指示と同じくらい、自分たち向けのメタデータ管理にも意識を向ける必要があります。

---

## まとめ

- Claude CodeはCLAUDE.mdのHTMLコメント（`<!-- ... -->`）をClaude自動注入時に除外するようになった
- この設計により「AIへの指示」と「開発者向けメモ」を同じファイル内で分離できる
- 逆に`WebFetch`はHTMLをMarkdownに変換してClaudeに渡す——AIにはMarkdown、人間のメタデータにはHTMLという一貫した棲み分け
- しかし「人間だけが読める層」は、意識して読む仕組みを作らないと誰も読まない
- HTMLコメントには期限を書き、定期的にgrepで棚卸しする習慣が効果的
